# QuaC — Data Quality Control Web App


This repository will be used to build a client-side TypeScript-based web app for data quality control (QC). The app would enable users to perform quality control checks and corrections, on their datasets, based on user-provided pre-specified QC rules. The app would also enable users to compose and serialize new QC rules or edit existing ones. The web app will be hosted on GitHub Pages. The app will entirely execute within the privacy of the user's browser. No data is sent to any remote server. The user is expected to load their own private datasets into this app, so data privacy assurance from the web app is critical. The codebase should be modular, organized, and easy to maintain.

Please devote a separate sub-agent to review my data table library (`@jeyabbalas/data-table`; https://github.com/jeyabbalas/data-table). It enables in-browser web apps to manage and display tabular data with a DuckDB-WASM backend. It provides users with interactive visualizations and filters for the data. DuckDB backend enables programmatic execution of SQL queries (for filters or QC rules) on the dataset. It also supports ways to annotate cells to highlight QC validation flags. It has features to neatly display JSON Schema column descriptions as tooltips in the column headers of the displayed data table. I think this library can be useful in this app to view the user input data, the output QC report generated by this app, and to compose or edit QC rules.

## Quality control rules

The quality control rules can be specified using two sets of artifacts: 1) JSON Schema file(s) to specify schema validation rules, and 2) CSV file(s) to specify QC rules other than schema validation—including data correction, semantic validation, dataset-level integrity rules, among others—that are unsuited for JSON Schema. Think of an informative name to refer to these non-schema validation QC rules files. While these files are designed to contain non-schema validation QC rules, they can technically also be used to specify schema validation rules. So, a generic name like QC rules file should be fine. In this prompt, I refer to them as non-schema validation QC rules to clarify their value proposition on top of JSON Schema validation. The JSON Schema, here, should always describe a single, tabular dataset as an array of objects. However, the JSON Schema may either be a single JSON file or a complex network of JSON files referencing each other. 

`tests/fixtures/hesp/json_schema/`: Contains an example of a complex JSON Schema network of files describing a single tabular Household Economic Security Panel (HESP) dataset as an array of objects.

`tests/fixtures/hesp/qc_rules/HESP_qc_rule_catalog.xlsx`: Contains a compiled list of example QC rules. "Non-JSON or ETL" sheet contains rules that in my opinion require a separate non-schema validation QC rules CSV file. Note that this specific file is not how I envision the format for the non-schema QC rules files in the app. This is meant purely for your edification. I am merely documenting and demarcating the schema validation rules (best specified using JSON Schema) from non-schema QC rules (not suited for JSON Schema), the way I see it. You are free to think beyond the confines of how I am thinking about this. My end goal is tabular data QC through rules.

These non-schema validation QC rules need to be specified using a CSV file. Please design a simple format for them. The format should be simple enough for the users to handcraft it themselves. 

Non-schema validation QC rules can be grouped into separate categories meaningful to the user, each can be confined to a separate CSV file. I think the CSV files could have the following columns: 
`rule_id`: Unique identifier for the QC rule. 
`rule_type`: Identify broad categories of QC rule types but do not be too specific. It should only serve to help the app process them. Document them all.
`rule_scope`: Scope of the rule (dataset-level, row-level, column-level, longitudinal, etc.). Examples of column-level assertions include: unique, no_nulls, monotonic, match_regex(r), in_enum([...]), count_distinct_in_range.
`target_variables`: List of variables affected by the rule. 
`condition`: IF clause of a rule, written in SQL. I think this can include DuckDB functions, which are supported by the `@jeyabbalas/data-table` library.
`update_target_values`: THEN clause when correcting/mutating target variable data using data correction rules, written in SQL (+DuckDB functions) or JavaScript. Perhaps a supporting column specifying the language used for this column `sql`|`sql+duckdb`|`js`. Most corrections are one-liners in SQL: TRIM(col), CASE WHEN col < 0 THEN NULL ELSE col END, type coerce, rescale. JS could be useful for regex capture groups, multi-step logic, external lib use. There should not be any THEN clause when merely flagging a validation error instead of modifying the data. Note that data privacy is critical for this app so the QC rules should be safely executed in the browser sandbox.
`comment`: Text to annotate data with human-readable explanation of the validation flag or what kind of correction was done and why.

The above columns are just suggestions. Feel free to design the non-schema validation QC rules files yourself. The design should be simple for the user to work with and unambiguous to the app.

The app should also have a tab enabling users to compose, edit, and serialize their own non-schema validation QC rules file(s). The `@jeyabbalas/data-table` library should help during their composition to see live the effects of each newly crafted QC rule. For schema validation rules, assume users are already well-versed with JSON Schema. Beyond basic checks to ensure that the JSON Schema is valid upon input, the app does not provide any utilities to compose, edit, and save them.

Please devote separate sub-agents to plan for the JSON Schema and the non-schema validation QC rules files. Both are quite complicated to figure out.


## App inputs

The app should accept 3 inputs from the user: 

A single tabular dataset file. Formats supported: JSON array of objects, CSV/TSV, Excel (users can choose sheet; default: sheet 1), Parquet.
JSON Schema file(s) specifying schema validation QC rules. If there are multiple JSON files referencing each other, write a logic to automatically detect the main JSON file. If it is ambiguous, use a modal to prompt the user to select the index file to resolve this. 
CSV file(s) specifying non-schema validation QC rules. These are named something generic in the app and codebase. The readme file encourages the use of JSON Schema for schema validation rules.

The inputs are loaded into the app either via manual file uploads or dereferenceable URLs. Dereferenceable URLs should enable automatic app configuration when users share links through URL parameters. There could just be a partly configured app, for example, only the QC rules files are specified via URLs but not the data. The user just needs to then upload the data to run validation on pre-specified and shared QC rules configuration. Here, I can share a URL to the app with pre-configured QC rules attached. If the index JSON file was resolved due to ambiguity, the selected index file ID should also be included in the URL params such that the receiver of the URL does not need to know anything about the JSON Schema structure.

The app must perform basic checks to ensure that the QC rules files are valid. When data is loaded into the app, perform a broad check to ensure that the data is pertinent to the QC rules files.

For your unit tests, it would be a good idea to create examples of these inputs in one of the early phases. For a real-world example, I have provided the JSON Schema for HESP dataset. Generate the non-schema validation QC rules and mock dataset (valid and invalid examples) for testing the app.


## App output

The app applies QC rules to the input dataset and generates a downloadable “QC report,” a multi-sheet Excel file. 

The first sheet of the QC report contains the original data annotated with all the validation flags and corrections made by the application of the QC rules. The annotations are in the form of sister columns inserted adjacent to the target column being flagged for review. In this annotation column, only the cells adjacent to the one being flagged for review, contains the comment text. The rest of the cells are blank. So, if the dataset has a variable `age`, the annotations can be contained in the `age__review`. If a variable did not raise any flags, no such column is inserted next to it. If multiple QC rules address the same target variable, the same annotation column is re-purposed and new text is appended to the existing comment. The text for the annotations come from the `$comment` key in the JSON Schema or `comment` column of the non-schema CSV rules file. If you can think of a way to elegantly represent all validation rules in both JSON Schema and non-schema validation QC rules file, please include them with the comment text. Foresee all possible validation rule structures, especially for JSON Schema. For non-schema validation QC rules, include rule ID for provenance.
Sheet 2 can list all the missing variables with their descriptions.
Sheet 3 can include validation flags that cannot be annotated with the data. This includes dataset-level checks.
Sheet 4 tallies repeat offenders to help users prioritize frequently occurring errors, which may indicate an underlying systemic issue.

Above, I am only describing what the downloadable QC report file looks like. You have more creative freedom in how to display this information in the web app. You do not need to display 4 sheets. You can display the original data using the `@jeyabbalas/data-table` library and annotate offending cells, columns, rows using the feature provided by the library. The information contained in the rest of the sheets can be displayed in an elegant UI. The above description is of the downloadable Excel file. It is a way to serialize this information. You again have creative freedom to improve upon my suggestion on how to organize and format the QC report. The only restriction is that it should be a single Excel spreadsheet.

Also, consider using the column header tooltips feature of the `@jeyabbalas/data-table` library to annotate each column with all the aggregated QC rules (JSON Schema or QC rules file) that are relevant to that column as a target.

Remember that `@jeyabbalas/data-table` does not allow you to mutate data, which is done by data correction QC rules. So, think through the data flow carefully.

The QC report is meant to be reviewed by the data stewards. This is part of the reason for such a portable and decentralized design of the app, where QC rules can be automatically loaded via URL parameters and sent to the data stewards to validate their data locally in the privacy of their browsers.


## App features

The app enables users to input the data, JSON Schema, and QC rules CSV file. It performs broad input verification. It checks if the JSON Schema and the QC rules files are valid. Resolves all ambiguities. When data is input, it checks if it is broadly pertinent to the input rules file. The app enables users to share pre-configured app via URL params, mainly used to pre-specify the JSON Schema and QC rules files.

The main function of the app is to run the QC rules and generate a downloadable QC report. Beyond the data correction rules, the app does not enable users to edit their data within the app. They are expected to fix their data then re-upload the data to see what errors remain. This is the main work surface area of the app.

The app should enable the user to compose or edit the QC rules file themselves. I think my data table library can be really beneficial here so they can see the effects of their newly composed rule, live. They can also review existing rules. I think CodeMirror can be used to provide assistance for composing this. Similar to my data table library, provide code completion and intelligence on what columns are available in the dataset. This will prevent common typographical errors. Enable users to check the rules before they save it into the QC rules file. The users can then download and persist the QC rules file to share.
